The Motivation

Why I built this

  • I bank with Investec. I open the app, check my balance, scroll through transactions, and close it. I never actually learn anything from it.
  • That got me thinking: what if my banking app actually understood me?
  • Not just numbers on a screen, but something that connects the dots, knows my habits, warns me before I overspend, and remembers my goals.
  • So I built it myself. Backend, frontend, AI, knowledge graph. ~10,000 lines of code, one developer.

▼
AI Model Layermodel.py: intent parsing + generation
▼

▼
Transaction Classifieragent.py: weighted fraud scoring
▼
Infrastructuremain.py: 41 routes, WebSocket
  • Voice parses natural language, manages multi-turn payment flows, stores memory to disk
  • AI Model extracts structured JSON intents with 28 types, falls back to keyword rules if offline
  • Knowledge Graph is the single source of truth. All balances, merchants, patterns live here

Lessons Learned

The hard parts

💻
Single-file Frontend
3,260 lines, 114 JS functions, zero frameworks. Managing state across a phone UI, WebSocket streams, D3 graph, and dashboard in vanilla JS. It works. But it wasn't pleasant.
🔎
Fraud Score Tuning
6 weighted indicators feeding a 0-to-1 score. Getting the weights right so it catches real threats without flagging every Woolworths run as suspicious took a lot of trial and error.
🎤
Voice Payment Flow
"Pay Mom R500" → extract name, fuzzy-match beneficiaries, disambiguate, extract amount, confirm, execute. All regex + state machine, no ML. Making it handle real speech was brutal.
⚡
Async Simulator
Background task firing transactions via WebSocket while UI, voice, and graph run concurrently. Race conditions everywhere. Got very familiar with asyncio debugging.

AI Model Choice

Why Qwen 2.5-3B?

  • 3 billion parameters: small enough to run on a laptop GPU
  • Local via Ollama: no cloud API costs, no rate limits
  • Data stays private: financial data never leaves the device
  • Fast inference: ~2-5s response time locally
  • Rule-based fallback: works even without the model running

A note to the API team

Building on Investec Open API

Thank you. The OAuth2 flow is solid, the sandbox is easy to start with, and real endpoints made this project possible.

WHAT WOULD'VE MADE IT EASIER
Real-time Transaction Events
Webhooks or event push. The sandbox is pull-only. I built an entire simulator just to get transaction streams. Push events would let agents react to real bank activity.
Transfer & Payment API
Sandbox is read-only. All write operations are simulated locally. Even a mock transfer endpoint returning success would help prototype payment flows.
Beneficiary API
Basic CRUD for beneficiaries. I hardcoded 7 demo beneficiaries. An API to list, add, manage them would enable real payment agent flows.
Richer Sandbox Data
Demo transactions are always the same. Rotating or randomized data would make testing feel much more realistic for developers.

Not complaints. A wishlist from someone who wants to keep building on this platform.
